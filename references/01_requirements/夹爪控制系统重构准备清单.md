# 夹爪控制系统重构准备清单

日期：2026-05-15

## 1. 重构目标

当前阶段已经证明 PC 到 USB2FDCAN 到达妙电机的基础链路可用。下一阶段重构的目标是把“联调可用代码”整理为“可维护、可验证、可移植的控制系统”。

重构后应满足：

- 控制算法、硬件访问、协议编码、UI 和日志分层清晰。
- 关键安全策略不依赖 UI 代码。
- 可在仿真、PC 实机测试、最终机器人控制器中复用同一套控制核心。
- 每个硬件后端都可以用同一接口替换。
- 参数和标定结果可配置、可追溯。
- 故障、堵转、接触、回零、卸载等行为可测试。

## 2. 需求文档建议拆分

后续整理需求时建议拆成以下文档：

| 文档 | 内容 |
|---|---|
| 产品需求 | 夹持对象、目标力、速度、寿命、环境、使用流程 |
| 机械约束 | 行程、机构模型、自锁、销轴、丝杆螺母风险 |
| 控制需求 | 状态机、限速限流、回零、夹紧、打开、防卡死 |
| 安全需求 | 最大电流、最大速度、温度、堵转、故障降级 |
| 电机和通信接口 | DM-J4310P-2EC 协议、ID、CANFD、SDK、驱动约束 |
| 标定需求 | 目标夹紧力到电流/力矩映射、摩擦、自锁、卸载量 |
| 测试验证 | 单元测试、仿真测试、样机测试、耐久测试、异常测试 |
| 软件架构 | 模块边界、线程模型、配置、日志、UI、部署 |

不要把“理论设计值”“样机测得值”和“临时调试参数”写在同一张表里。建议每个参数都标注来源：

```text
design: 理论或结构设计值
measured: 样机实测值
calibrated: 标定后软件使用值
temporary: 调试临时值
```

## 3. 目标软件架构

建议分层如下：

```text
app/
  gripper_app              调度、生命周期、配置加载

control/
  gripper_controller       状态机和控制策略
  safety_limiter           限流、限速、限温、软限位
  contact_jam_detector     接触和堵转检测
  force_mapper             夹紧力到电流/力矩映射
  friction_identifier      摩擦识别

motor/
  motor_interface          电机抽象接口
  simulated_motor          仿真后端
  damiao_motor             达妙电机对象

protocol/
  damiao_protocol          帧编码、帧解析、寄存器读写

transport/
  can_bus_interface        CAN 抽象
  dm_usb2fdcan_bus         达妙 DeviceSDK 后端
  python_can_bus           PC 调试后端，可选
  replay_bus               日志回放测试后端

ui/
  test_ui                  PC 测试界面，只调用 app/control 接口

config/
  gripper_config           机构、电机、限幅、标定参数

logging/
  structured_logger        状态、命令、反馈、CAN trace
```

核心原则：

- `control` 层不能直接依赖 Tkinter、ctypes、DeviceSDK。
- `protocol` 层不能知道夹爪状态机。
- `transport` 层只负责收发帧，不解析夹爪业务。
- `ui` 层不能实现安全兜底，只能显示和触发命令。

## 4. 当前代码迁移建议

### 4.1 Python UI

当前文件：

```text
src/gripper_control/ui/gripper_control_ui.py
```

里面混合了：

- UI 控件
- 控制状态机
- 仿真电机
- 达妙协议
- DeviceSDK ctypes 绑定
- CAN trace 和诊断

建议迁移顺序：

1. 提取 `DamiaoProtocol` 到独立模块。
2. 提取 `MotorBackend`、`SimulatedMotorBackend`、`DamiaoUsb2FdCanBackend` 到 `motor/transport` 模块。
3. 提取 `GripperControllerSim`，改名为 `GripperController` 或 `PythonGripperController`。
4. UI 只保留参数输入、按钮、状态显示和日志显示。
5. 将默认参数集中到配置文件，不在 UI 控件构造函数中散落。

### 4.2 C++ 控制骨架

当前 C++ 目录已经具备较好的模块雏形：

```text
src/gripper_control
```

已有模块：

- `GripperController`
- `SafetyLimiter`
- `ContactJamDetector`
- `FrictionIdentifier`
- `ForceCommandMapper`
- `MotorInterface`
- `DmJ4310Protocol`
- `DmJ4310Motor`
- `SimulatedMotor`

建议下一步：

1. 把 Python UI 中已经验证的实机修正同步到 C++：
   - 首次强制写控制模式。
   - 未归零前不自动把第一帧反馈设为零点。
   - 未归零时用小步相对目标回零。
   - CANFD/BRS 作为可配置能力。
2. 实现 `DmUsb2FdCanBus`，封装 `DM_DeviceSDK v1.1`。
3. 将 `DmJ4310Motor` 的停止、使能、模式切换加入寄存器回读确认。
4. 增加全行程位置绕圈处理或明确电机反馈是否已经是输出端多圈位置。
5. 用同一组协议单元测试覆盖 Python 和 C++ 的帧编码。

## 5. 必须保留的控制行为

重构时以下行为不能丢：

- 全行程闭合都必须限速限流。
- 0-12 mm 区间也要接触检测和堵转检测。
- 回零只能低速低电流。
- 夹紧接触后不能继续用大位置目标硬顶。
- 目标夹紧力必须斜坡建立。
- 夹紧完成后必须先卸载，再失能。
- 打开前必须低电流解锁。
- 堵转和无响应都必须主动停止。
- 任何状态下绝对电流、速度、温度限制必须优先级最高。
- 通信异常、反馈超时、驱动故障必须进入安全状态。

## 6. 标定和参数体系

建议将参数分为四类。

### 6.1 机构参数

```yaml
stroke_min_mm: 0.0
stroke_max_mm: 16.0
screw_lead_mm: 2.0
stroke_sign: 1
```

### 6.2 电机和通信参数

```yaml
motor_id: 0x08
master_id: 0x18
control_mode: pos_force
canfd: true
bitrate: 1000000
data_bitrate: 5000000
max_phase_current_A: 20.0
torque_per_amp_Nm: 0.625
```

### 6.3 安全限幅参数

```yaml
close_speed_mm_s: 0.30
open_speed_mm_s: 0.80
homing_speed_mm_s: 0.20
max_current_A: 2.00
homing_current_A: 0.35
unlock_current_A: 0.40
temperature_fault_C: 75.0
```

### 6.4 标定参数

```yaml
friction_close_A: measured
friction_open_A: measured
force_current_curve: calibrated
unload_mm: calibrated
contact_current_delta_A: calibrated
stall_speed_threshold_mm_s: calibrated
```

标定参数必须来自样机测试，不建议长期使用当前 UI 中的占位公式。

## 7. 测试矩阵

### 7.1 软件单元测试

| 测试 | 目的 |
|---|---|
| 达妙帧编码 | 确认 enable、disable、POS_FORCE、MIT、寄存器读写帧 |
| 反馈解析 | 确认 ID、错误码、位置、速度、力矩、温度 |
| 限幅器 | 确认电流、速度、行程、温度裁剪 |
| 状态机 | 确认回零、夹紧、卸载、打开、故障流程 |
| 接触检测 | 覆盖不同摩擦、不同速度、不同电流噪声 |
| 防卡死 | 覆盖打开失败、闭合异常堵转、无反馈 |

### 7.2 仿真测试

| 场景 | 预期 |
|---|---|
| 空载全行程闭合 | 到达软限位并报未接触 |
| 0-12 mm 夹持对象 | 低速接触检测生效 |
| 14-28 mm 线缆 | 正常接触、建力、卸载、失能 |
| 高摩擦 | 自检识别摩擦升高或进入降额 |
| 打开卡滞 | 低电流解锁，失败后进入防卡死 |
| 通信丢帧 | 反馈超时进入安全状态 |

### 7.3 样机测试

| 测试 | 输出 |
|---|---|
| 空载摩擦识别 | 开向/合向电流分布 |
| 夹紧力标定 | 目标力和电流/力矩曲线 |
| 卸载量标定 | 保持力、解锁成功率、抱死概率 |
| 不同线径夹紧 | 14/20/28 mm 下的接触行程和力 |
| 其他物体夹持 | 0-12 mm 任意接触的冲击和停止 |
| 重复寿命 | 丝杆螺母温升、磨损、卡滞概率 |
| 异常断电 | 夹持保持和再次打开能力 |

## 9. 重构里程碑

### M1：整理需求和参数

- 输出正式需求文档。
- 确认硬件 ID、通道、CANFD 配置。
- 确认默认安全限幅。
- 明确力控精度目标和验收方法。

### M2：抽离 Python 模块

- UI 和控制逻辑分离。
- 协议和硬件后端分离。
- 默认参数改为配置文件。
- 保持当前 UI 功能不退化。

### M3：C++ 硬件后端

- 实现 `DM_DeviceSDK v1.1` 的 C++ CAN bus 封装。
- 同步 Python 中已验证的模式切换、CANFD、小步目标策略。
- C++ 单元测试和硬件 smoke test 通过。

### M4：标定和安全验证

- 完成摩擦识别标定。
- 完成夹紧力曲线标定。
- 完成卸载量标定。
- 完成异常堵转和恢复策略验证。

### M5：产品化接口

- 对接机器人主控。
- 定义命令接口和状态反馈接口。
- 固化配置加载、日志、错误码。
- 完成样机耐久测试。

## 10. 下一轮需求整理问题

后续重新整理需求时需要明确：

1. 最终控制程序运行在 PC、机器人主控，还是嵌入式控制器？
2. 最终软件语言以 C++ 为主，还是 Python 只保留测试工具？
3. 夹紧力精度目标是多少，是否允许只做电流估算？
4. 夹紧完成后失能保持的最小保持力和保持时间要求是多少？
5. 丝杆螺母抱死的判据和允许恢复动作等级是什么？
6. 是否需要保存每次夹紧日志用于追溯？
7. 现场是否允许人工重新回零，还是必须全自动恢复？
8. 电机反馈位置是否在全 16 mm 行程内连续无绕回？
9. 不同线缆外皮材料、硬度、椭圆度是否要分别标定？
10. 最终 UI 是调试工具，还是产品交付工具？
